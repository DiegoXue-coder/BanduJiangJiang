# 后端 API 设计：从数据库到"能做的事"

> 学习背景：架构规划的最后一块。理解"数据库设计"和"API 设计"是两个不同的层面，
> 再用一个真实使用场景倒推出伴读讲讲需要哪些 API 动作。

## 核心区分：数据库设计 vs API 设计

- **数据库设计**关心"东西怎么存"——名词：用户、书、划线、问答记录
- **API 设计**关心"能让服务器做哪些事"——动词：问问题、存划线、查书本信息

`/ask`、`/tts`、`/context/{book_id}` 这些不同的接口名字，代表的是手机 App 能让服务器执行的不同**动作**，不是数据库分类。这是两个相关但独立的设计层面，容易被混为一谈。

## 用真实使用场景倒推出的动作清单

跟着"打开 App → 看书架 → 翻开一本书 → 读到一段划线+提问 → 听语音朗读"这个场景走一遍，倒推出：

| 动作(动词) | 对应场景 | 涉及的表 |
|---|---|---|
| `import_book(epub文件)` | 第一次把书导入书库（容易被漏掉的起点） | 写 `books` + `chapters` |
| `get_library()` | 打开 App 看书架 | 读 `books` + `reading_progress` |
| `get_book_context(book_id)` | 点开书，跳到上次读到的位置 | 读 `books` + `chapters` + `reading_progress` |
| `save_highlight(...)` | 划线 | 写 `highlights`（含 embedding） |
| `ask_about_passage(question)` | 问 AI 问题 | 向量检索 `qa_history`/`highlights` → 组装上下文（RAG）→ 调 DeepSeek → 写 `qa_history` |
| `text_to_speech(text)` | 听语音朗读 | 不碰数据库，纯调用 TTS |

## 设计原则：呼应"两条腿走路"

每个动作都按"独立能力"设计——`save_highlight` 不管是手机 App 调用，还是以后包一层 MCP 给 Claude/豆包这类 AI 助手调用，都是同一个动作、同一套逻辑，不需要为不同调用方重新设计。这是 [[project_bandujiangjiang_mobile_pivot]] 里"两条腿走路"决策的具体落地。

## 排查方法：检查"起点"覆盖到没有

梳理功能清单时，容易只顺着"熟悉的中间流程"走一遍就以为齐全了（比如默认"书已经在书架上"），却漏掉"这个状态最初是怎么来的"这类起点动作（比如 `import_book`）。以后自己梳理任何功能流程，要额外检查一下"最开始的起点"是否也被覆盖到。
