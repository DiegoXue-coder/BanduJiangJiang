## 产品架构策略

- ✅ **Build vs Integrate（自建 App vs 寄生 AI 生态）**——理解 MCP/Skill 这类"AI 插件"架构能降低分发摩擦，但不会消除"如何获取上下文"这个问题；理解为什么后端接口要按"独立能力"设计而不是为某个 UI 页面定制。

## 移动端开发

- ✅ **跨平台开发框架选型（React Native vs Flutter）**——理解 iOS(Swift)/Android(Kotlin) 各自"母语"不同，两个框架用不同机制解决"写一份代码两边跑"的问题（RN 借用原生组件 vs Flutter 自己画像素）；能自己总结出"要不要分子级跨平台一致性"这个选型判断标准，并据此判断伴读讲讲该用 React Native。详见 [02-跨平台移动开发框架选型.md](02-跨平台移动开发框架选型.md)。

- ✅ **EPUB 书库：格式/分页/划线定位/渲染方案**——理解 EPUB 本质是打包的 HTML 章节文件+索引；理解"分页"是阅读器现场计算、无统一标准；理解 CFI（结构路径地址）如何让划线跨字号/跨设备稳定定位；理解 React Native 里用 WebView + epub.js 渲染电子书的"取巧"方案，以及它带来的性能开销和桥接复杂度代价（尤其是 AI 讲解/TTS 需要跨这座桥获取当前段落）。最终选定 `@epubjs-react-native`。详见 [03-EPUB书库渲染与划线定位.md](03-EPUB书库渲染与划线定位.md)。

- ✅ **Expo Go（通用车）vs 自定义开发版本（自己造车）**——理解 Expo Go 是预装好常用原生零件的通用容器 App，适合大多数库；一旦某个库需要 Expo Go 没预装的原生零件（如 `react-native-readium` 需要的原生 Readium 引擎），就必须切换到"自定义开发版本"，要走真正的原生编译流程，是一次性的环境搭建成本。这是判断"任何新 npm 库要不要额外开发成本"的通用标准。详见 [04-Expo-Go与自定义开发版本.md](04-Expo-Go与自定义开发版本.md)。

## 产品策略（补充深挖）

- ✅ **内容来源：DRM 与"公版经典"定位**——理解 DRM（数字版权管理）如何锁死 Kindle/微信读书/苹果图书购买的商业电子书，无法合法导入第三方阅读器；只有公版书（版权保护期已过）和用户自有 DRM-free 文件能自由导入。将这个限制转化为产品定位——聚焦"公版经典/值得精读的书"而非追当代畅销书，v1 可以先做预置书库而非通用导入功能。

- ✅ **账号体系：v1 最小可行方案**——理解认证要解决的两个不同问题（防盗用 vs 区分用户身份），用"会员卡 vs 身份证"类比区分两种验证机制；确定 v1 方案是复用现有 HMAC 会员卡式验证（从 Chrome 插件搬到手机 App），`user_id` 写死为固定值，不做登录注册界面；数据库层面已提前铺好 `users` 表，未来扩展多用户时只需加验证层，不用动数据存储设计。详见 [07-账号体系设计.md](07-账号体系设计.md)。

- ✅ **后端 API 设计：数据库设计 vs API 设计的区分**——理解"数据库设计管东西怎么存（名词），API 设计管服务器能做哪些事（动词）"这个核心区分；用真实使用场景（打开App→看书架→翻书→划线提问→语音朗读）倒推出完整动作清单（`import_book`/`get_library`/`get_book_context`/`save_highlight`/`ask_about_passage`/`text_to_speech`）；理解每个动作要按"独立能力"设计，呼应"两条腿走路"决策，为未来 MCP 化做准备。详见 [08-后端API设计.md](08-后端API设计.md)。
