# 语音对话架构流派与"实时打断"的本质

> 学习背景：为伴读讲讲手机端设计语音交互时，最初想选"实时打断式对话"，
> 通过讨论发现这本质上不是前端框架（React Native / Flutter）能决定的事，
> 而是"AI 大脑用哪个模型"决定的事。这篇笔记记录当时想清楚的过程。

## 两大流派

### 流派一：级联式（Cascaded）—— 伴读讲讲现在用的架构

```
音频 → 语音识别 STT → 文字 → 大语言模型 LLM → 文字 → 语音合成 TTS → 音频
（SenseVoice）        （DeepSeek）              （edge-tts）
```

- 每一步是独立模型/服务，像流水线拼起来。
- 优点：模块化，随时换供应商，成本低，兼容任何文字 LLM。
- 缺点：
  - 延迟是各段时间的总和。
  - 语音一旦转成文字，语气/停顿/情绪/呼吸全部丢失，LLM 读到的只是文字。
  - "用户是否说完了"只能靠 VAD（停顿检测）这种统计学猜测，分不清"思考中的停顿"和"真说完了"。

### 流派二：端到端语音模型（Native Speech-to-Speech）

代表：OpenAI Realtime API、豆包实时语音大模型、通义 Qwen-Omni-Realtime。

```
音频 → [单一模型，直接在音频上训练/推理] → 音频（持续双向流式传输，WebSocket/WebRTC）
```

- 模型本身在海量音频数据上训练，不是"文字模型 + 外挂语音"。
- 能听出语气/犹豫/呼吸，"语义级 VAD" 判断打断意图准得多。
- 音频双向持续流动，可以边"说"边"听"，听到打断信号立刻停止生成。

## 护城河：为什么不是工程量大小的问题

个人/小团队想自己做出"端到端实时打断"，会同时卡在三件事上，不是多写代码能补的：

1. **训练数据规模** —— 需要海量真实"人类对话打断"音频数据训练模型判断轮次，这是数据量级壁垒。
2. **回声消除（AEC）** —— 手机外放时麦克风会录到 AI 自己的声音，系统要分清"这是自己的回声"还是"用户真的在插嘴"，是声学工程 + 模型联合调优的老问题。
3. **低延迟全球流式基础设施** —— 实时音频编解码 + GPU 流式推理服务，是系统工程投入。

**即便是豆包这种大厂**，公开资料里也提到实时语音的打断"还有点小瑕疵，需要提高音量才能打断"——说明 2026 年这个领域本身仍在成熟期，不是个人做不好，是大家都还没做到完美。

## DeepSeek 现状

DeepSeek 目前只有文字 API，没有语音模式（不是端到端语音模型）。这意味着伴读讲讲现在的架构（DeepSeek 文字 + edge-tts + SenseVoice）天生属于"级联式"，无法在不换核心大脑的前提下做出真实感的"实时打断"。

真想要端到端实时打断，只有两条路：
- 接入豆包实时语音大模型或 Qwen-Omni-Realtime 作为语音对话部分的大脑（国内这两家目前要求企业实名认证/备案才能稳定调用，个人开发阶段会卡住）
- 自己拼一套连续监听 + 回声消除 + 打断逻辑（连大厂做的都有瑕疵，个人从零拼难度更大）

## 最终决策（2026-07-06）

伴读讲讲 v1 采用**级联式架构 + 一键打断按钮**：
- AI 说话时提供一个"停止播放"按钮，点一下就中断当前 TTS 播放，可以立刻问下一个问题。
- 工程上只是播放器的 `stop()`，不需要连续监听麦克风、不需要回声消除。
- 保留 DeepSeek 作为文字大脑，现有 STT/TTS 管道不用大改。

**未来升级路径（按复杂度递增）：**
1. 流式分句 TTS —— LLM 边生成文字边按句子切给 TTS，不等整段生成完，依然是级联架构但体感更快。
2. 换成端到端语音大模型（豆包/Qwen-Omni-Realtime）—— 才是真正的"实时打断"，但涉及企业认证、协议重新设计、成本上升，等有真实用户反馈需要时再考虑。

## 关联决策

这次讨论也让我们重新想清楚了：伴读讲讲真正的差异化价值是"AI 用语音把内容讲出来" + "苏格拉底式提问设计" + "阅读内容沉淀成数据飞轮"，而不是"打断"这个交互细节本身。语音输入端的"不打断心流"问题，市面上的语音输入法（如讯飞输入法）已经解决了，不需要重新造轮子。
