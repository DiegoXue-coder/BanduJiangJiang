# 数据库 Schema 设计：数据飞轮的地基

> 学习背景：为"数据飞轮"（跨书连接、读书复盘等）设计数据库结构。用户已有关系型数据库/
> CRUD/星型模型/物化视图基础，这次讨论直接在专业术语层面进行，聊的是"这个产品具体需要
> 哪些实体"和几个真实的设计取舍，而不是数据库基础概念。

## 现状：已有的比想象中多

`api/main.py` 已经有 `qa_history` 表并接入了 **pgvector**（PostgreSQL 向量扩展），每条问答存了 `embedding vector(1024)`（用 BAAI/bge-m3 模型算的语义向量），并且已经有真正在跑的语义相似度检索代码。这已经是一个初级的数据飞轮雏形，不是从零开始。

**发现的问题**：`qa_history` 有一行 `DELETE FROM qa_history WHERE created_at < NOW() - INTERVAL '90 days'`——为 Chrome 插件场景设计的自动清理，跟"数据飞轮要长期积累"直接冲突。

Chrome 插件里"苏格拉底提问"已经做出来了，是 `/ask` 接口的 `style="socratic"` 参数，靠专门的提示词实现，属于提示词工程层面，手机端可以直接复用这套逻辑，不需要重新设计架构。

## 最终 Schema（实时业务层）

| 表 | 关键字段 | 说明 |
|---|---|---|
| `users` | `id`, `created_at` | 现在单用户也先建，所有其他表挂 `user_id` 外键，避免以后多用户时重构所有表 |
| `books` | `id`, `user_id`, `title`, `author`, 文件位置, `added_at` | 自建 EPUB 书库的元数据 |
| `chapters` | `id`, `book_id`, `order_index`, `title` | 把 EPUB 文件自带的章节索引复制一份进数据库，换查询速度（反范式化的典型取舍：不用每次都重新解压解析 epub 文件查章节信息） |
| `highlights` | `id`, `user_id`, `book_id`, `chapter_id`, **`cfi_location`**（结构地址，见 EPUB 笔记）, `highlighted_text`（冗余存一份原文）, `note`, `embedding vector(1024)` | 核心表，划线定位靠 CFI 而不是页码，`embedding` 用于跨书连接 |
| `reading_progress` | `user_id`, `book_id`, `current_cfi_location`, `updated_at` | 每本书一行，记录"读到哪了"，更新型数据不是流水账 |
| `qa_history`（已有，需改造） | 加 `user_id`；**去掉 90 天自动删除** | 永久保留，支撑数据飞轮 |

## 决策点与取舍（已确认）

1. **建 `chapters` 表**——反范式化换查询速度：不用每次都重新解压解析 epub 文件查章节信息，代价是多一张表要维护、导入书时要多写一步解析索引存进去的逻辑。
2. **`qa_history` 取消 90 天自动删除，永久保留**——数据飞轮的前提就是数据不能丢。运维顾虑（没有专属服务器、数据库运维）在单用户阶段不是真问题：几年下来数据量顶多几万行，Railway 托管的 Postgres 完全扛得住，备份维护平台已经处理了。这个问题只有在有了成百上千真实用户之后才需要认真解决，现在解决是"过早优化"。
3. **跨书连接（向量相似度检索）选现算，不做预计算缓存**——两者的取舍：
   - **现算**：用户请求时当场跑 pgvector 相似度查询。好处是数据永远最新、不用维护额外后台任务；坏处是每次查询要付计算开销——但在单用户、数据量小（几千条划线）的阶段，这个开销小到可以忽略，配合索引几毫秒就查完。
   - **预计算缓存（类似物化视图）**：后台定时批量算好结果存进缓存表。好处是查询几乎零成本；坏处是有滞后（刚划的线要等下次批量计算才会被关联到），而且多一套定时任务要建要维护。
   - **结论**：现阶段选现算，等真遇到查询变慢的具体证据，再加物化视图这层加速——到时候能对症下药，而不是现在预先猜测。

## 设计哲学

**先把 OLTP 核心表规范化设计好，分析层的具体加速手段（物化视图、星型模型式的汇总表）留到有明确分析需求时再加**——这样今天的地基足够干净，以后任何新的分析需求都是往上加东西，不需要回头重构底层表结构。这跟前面几次技术选型（语音交互、EPUB 渲染库）的判断逻辑一致：不为不存在的问题提前买单。
