# Expo Go（通用车）vs 自定义开发版本（自己造车）

> 学习背景：在挑选 EPUB 渲染库时，发现候选之一 `react-native-readium` 需要"跳出 Expo Go
> 工作流"，由此引出这个更通用的移动开发概念——以后凡是用到"需要原生模块"的库，都会碰到同样的判断。

## 核心类比：通用车 vs 自己造车

**Expo Go**：Expo 公司提前造好的一辆"通用车"，发布在苹果/安卓应用商店，谁都能下载。车里已经预装了一堆常用零件（摄像头、录音、文件读写等）。写代码时用 `npm run tunnel` 这类命令，本质是把 JS 代码（"开车指令"）通过网络发给手机上这辆**已经造好**的车，车子照着指令跑起来——完全不需要自己"造车"，这就是为什么日常开发这么方便、扫码即用。

**自定义开发版本（custom development build / dev client）**：如果想用的某个零件压根没被预装在这辆通用车里（比如原生 Readium 引擎），没法打开 Expo 公司已经造好的成品车往里塞新零件——必须自己重新走一遍真正的"造车"流程（专业说法：编译/构建），造出一辆只属于自己项目、装了这个特殊零件的 App，装到手机上替代通用的 Expo Go，专门测试用。

## 判断标准

以后遇到任何一个新的 npm 库，可以用这个问题判断开发成本：**这个库需不需要额外的原生零件（不在 Expo Go 预装的常用零件列表里）？**

- 不需要 → 继续用 Expo Go，开发方式不变（本项目现阶段大多数功能属于这一类）
- 需要 → 意味着要切换到自定义开发版本，多一次性的原生环境搭建成本（iOS 要碰 Xcode/Podfile，Android 要碰编译版本配置），以后项目改动往往还要重新"造车"一次

## 术语补充

- **npm（Node Package Manager）**："别人写好的代码的应用商店"，全世界程序员把可复用的代码块发布在这里，`npm install xxx` 就是把别人写的代码下载进自己项目。
- **Readium**：一个专业的原生 EPUB 渲染引擎（不是"瑞典"，是语音输入的同音误听），`react-native-readium` 库包的就是它。

## 关联决策

EPUB 渲染库选型（详见 [03-EPUB书库渲染与划线定位.md](03-EPUB书库渲染与划线定位.md)）最终选择 `@epubjs-react-native`，正是因为它不需要额外原生零件，可以继续留在 Expo Go 工作流里，不用现在就承担"造车"的一次性成本。
