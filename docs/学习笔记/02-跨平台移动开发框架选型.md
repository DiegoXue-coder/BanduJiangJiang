# 跨平台移动开发框架选型：React Native vs Flutter

> 学习背景：为伴读讲讲手机 App 选技术栈时，从"为什么苹果和安卓不能装同一份代码"这个
> 生活化问题一路推导到 React Native / Flutter 的机制区别，最后自己判断出结论。

## 起点：手机系统天生"听不懂"同一种语言

- 苹果手机(iOS)的母语是 **Swift**
- 安卓手机的母语是 **Kotlin**
- 两者都不是 JavaScript——写 Chrome 插件用的 `content.js` 那门语言，两个手机系统都不天生认得

历史上想"规规矩矩"同时支持两个平台，程序员真的要写两遍：学 Swift 写一份给苹果，学 Kotlin 写一份给安卓，两套代码两套 bug，永远同步维护。

## 解决方案：跨平台"翻译层"框架，两大流派

**React Native（Meta/Facebook 做的）**
用 JavaScript 写代码，最终"翻译"成调用手机里**真正原生**的按钮、滚动条等零件。生成的 UI 就是苹果/安卓自己原生的组件，只是"由谁下指令"变成了你的 JS 代码。

**Flutter（Google 做的）**
用一门新语言 **Dart** 写代码，但不找手机借零件——自己把屏幕上每个像素都画出来，包括按钮的样子，完全不依赖 iOS/安卓的原生组件。

## 类比：奶茶店开分店

想象要在北京和纽约同时开奶茶店，想让两地喝起来一样：

- **方案 A（≈ React Native）**：把配方教给当地加盟商，用当地设备和原料做。地道，但两地设备原料终究有细微差异，做出来的东西可能不是分子级一样。
- **方案 B（≈ Flutter）**：自己造一整套完全相同的设备和原料运到两地，店员只负责照做。分子级一样，但代价是要自己把整套设备原料从头造出来、运过去，不能借用当地任何现成的东西。

（补充：两个框架其实都能做到"两边一样"或"两边不一样"，区别只在于默认情况下哪个方向更省力——RN 默认借用当地零件所以天生有点差异，Flutter 默认自己画所以天生分子级一致；反过来定制则要多花功夫。）

## 决策框架（自己总结出来的）

产品是否需要"分子级"的跨平台一致性？

- **需要**（比如竞技类游戏，两边操作/界面必须绝对一致，否则不公平）→ 选 Flutter
- **不需要，甚至平台间略有差异更好**（贴合各自平台的使用习惯和审美预期）→ 选 React Native

## 结论：伴读讲讲选 React Native / Expo

三个理由汇成一个答案：

1. 伴读讲讲是"AI 语音伴读"类阅读 App，不是竞技游戏，不需要分子级跨平台一致性，苹果/安卓用户各自贴合平台习惯即可。
2. 已经会 JavaScript（写过 `content.js`、`popup.js`），React Native 直接复用这门语言，Flutter 则要从零学 Dart。
3. `mobile/` 目录已经有一个用 React Native + Expo 搭的原型（`App.js`），已经跑通对话链路，不是从零开始。

**不切换 Flutter，继续在现有 React Native/Expo 原型基础上开发。**
