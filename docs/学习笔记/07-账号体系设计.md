# 账号体系设计：v1 阶段的最小可行方案

> 学习背景：之前"先单用户"是快速选择，没有深挖。这次补充讨论了认证机制的两个不同目的，
> 以及现有 Chrome 插件的 HMAC 机制怎么原样搬到手机 App 上。

## 认证要解决的两个不同问题

1. **防止陌生人盗用服务**（不管是谁，只要不是合法调用方就该被挡在外面，防止白嫖 DeepSeek/TTS 调用额度）
2. **区分"这是哪个具体的人"，把数据正确挂到对应用户身上**（长期看，是用户分析、算法建模、数据隔离的关键）

伴读讲讲现在单用户阶段，真正需要解决的是①，②可以等以后多用户时再做——但数据库层面已经提前建好 `users` 表（哪怕现在只有一行数据），为将来的②做好了铺垫。

## 两种认证的类比

- **会员卡式验证（现在需要的）**：像健身房检查会员卡，只要卡是真的就放行，不需要知道具体是谁。现有 Chrome 插件的 HMAC 每日令牌机制就是这种——只证明"这是我自己做的正牌客户端发来的请求"，不区分具体用户。
- **身份证式验证（以后多用户才需要）**：像银行开户，要验证"你确实是张三本人"，需要密码/验证码/第三方登录，还要维护登录状态。复杂得多，现在不需要。

## v1 最终方案

- 沿用现有 HMAC"会员卡式"密钥验证，把同一套机制从 Chrome 插件搬到手机 App（密钥写进 App 代码里，每次请求自动算"当日暗号"贴上，后端校验）——**不是重新设计，是复用+换个载体**。
- 数据库里 `user_id` 现在写死为固定值（如 `1`），不做任何登录/注册界面。
- 用户使用体验：打开 App 直接用，没有登录页、注册页、忘记密码、退出登录——因为这些功能现在根本不存在。
- 未来扩展路径：真正有多用户需求时，只需要加登录/注册这一层（验证"是谁"），书籍/划线/数据怎么存的底层设计完全不用动，因为 `user_id` 这一列从数据库设计阶段就已经铺好了。
