# 验收标准（Acceptance Criteria / Definition of Done）

> 背景：2026-07-14 用户在阶段四真机测试时发现整个项目从未定义过"做到什么程度算通过"，
> WBS 只写了任务范围没写验收门槛（规划层的真实疏漏，已如实留档）。自阶段四起，
> **每个阶段开工前必须先在本文档定好验收标准，缺标准不开工。**
> 阶段一到三已事实性收尾，经用户确认不回溯补写。

## 通用完成定义（Definition of Done）——适用于每一次交付，与具体阶段无关

任何功能或修复，**同时满足**以下各条才算"完成"（阶段验收标准是在这个底线之上叠加的）：

1. 真机验证过，不是只在 web 端/本地跑通（对应经验法则第6条"验证分层如实汇报"）
2. 代码已推送且 Railway 自动部署成功、`/health` 正常（本项目合并即上线）
3. 改动已写进开发进度记录（做了什么、为什么这么做）
4. 对 `node_modules` 的任何修改有 patch-package 补丁持久化
5. 不破坏 `extension/` 依赖的旧接口和旧表结构（"只加不改"原则）
6. 对照当前阶段验收标准自检过，汇报时说明哪些项已自测通过、哪些需要用户验证

> 颗粒度说明：质量管理按三层设计——本 DoD 是全局底线（一次定义处处生效）；
> 阶段验收标准只给交付物定（下方各节）；任务级质量不写文档，未来靠自动化测试实现
> （单人+小代码量阶段人工验收够用，代码量增长或引入协作者时再补后端 API 自动化测试，已记入规划）。

## 阶段四：AI 对话 + 语音

### 功能验收（全部必须在真机上验证，web 端不算数）

1. **上下文相关性**：长按选字→"问AI"，AI 回答必须紧扣所选文字和这本书的内容（不是脱离原文的泛泛而谈）——《道德经》《论语》各抽查 3 处不同段落验证
2. **两种回应模式都要**（2026-07-14 用户拍板）：直接讲解模式 + 苏格拉底提问模式，用户可切换；复用插件版已验证的 style 参数提示词，苏格拉底模式必须"先提问引导，不直接给答案"
3. **对话串 UI**：多轮对话堆叠可滚动，AI/用户气泡视觉区分，追问时上下文连贯（AI 记得前几轮说过什么）
4. **TTS 播放 + 手动打断**：回答可语音播放；停止按钮点击后**即刻静音**（这是"级联式+手动打断"架构决策的最终落地验收点）
5. **STT 语音输入**：麦克风按钮录音→转文字→正常发送提问
6. **数据飞轮第一环**：每次问答写入 `qa_history` 且带 embedding（用数据库查询核对，不只看界面）；划线时的 embedding 计算已在阶段三实现，此处验证问答侧
7. **RAG 上下文注入**：提问时向量检索历史相关问答/划线并注入 prompt（按 WBS 阶段四范围）

### 质量门槛（2026-07-14 用户拍板）

- **响应速度**：非流式返回 + 等待期间显示加载动画，从发送到完整显示 **10 秒内**为通过；流式输出明确排除在 v1 之外，留作后续优化
- **回答长度**：沿用插件版 System Prompt 的字数约束（TTS 朗读时长合理）

### 健壮性

- 断网 / 后端不可用时有明确报错提示，**不允许无提示卡死**（阶段三选区菜单 bug 的同类问题引以为戒）
- AI 接口调用失败不影响划线、阅读等其他功能继续使用（Promise.allSettled 原则）

### 阻塞项（验收前必须先修复）

- **选区手柄拖动 bug**：长按选字后拖动手柄调整范围，操作菜单消失且不再出现——用户无法精确划选想问的文字，直接阻塞"问AI"核心流程，此 bug 不修复阶段四不能验收

## 阶段五：划线复盘页面

> 2026-07-17 草拟并经用户拍板确认，正式生效。

### 功能验收

1. **跨书聚合列表**：一个列表里混排展示用户在所有书里的划线（`highlights`）和问答记录（`qa_history`），不分书本 tab，按时间倒序（最新的在最上面）
2. **列表条目信息**：每条至少显示——书名、划线原文或提问内容、时间。问答记录额外显示 AI 的回答（哪怕摘要/截断显示）
3. **点击条目不跳转**（2026-07-17 用户拍板）：v1 只做纯列表展示，不做跳回阅读器原位置的 CFI 定位跳转，留到以后需要时再做
4. **独立入口**：底部导航"划线复盘" tab（已搭好），不依附书架页面
5. **空状态**：一本书都没读、或读了但没有任何划线/问答时，要有清晰的空状态提示（不能是空白页或卡死）

### 质量门槛

- 列表首屏加载 **3 秒内**完成（比对话式交互的 10 秒门槛更紧，因为这是纯读取展示，没有 AI 调用）
- **不做分页/虚拟滚动**（2026-07-17 用户拍板）：目前数据量级小（2本书、单用户测试数据），一次性加载全部即可，等以后数据量真的变大、加载明显变慢时再补分页，不为不存在的问题提前买单

### 健壮性

- 加载失败（断网/接口报错）要有明确提示 + 重试入口，不能无提示卡死（跟阶段四同样的门槛）

### 后端改动范围（只加不改）

- 新增一个跨书聚合接口（暂定 `GET /app/review`），按 `user_id` 查 `highlights` + `qa_history`，按 `created_at` 倒序合并返回；不改动现有 `/app/books/{id}/highlights` 等接口
