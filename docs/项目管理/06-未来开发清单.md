# 未来开发清单（Backlog）—— 伴读讲讲手机端

> 跟 `03-风险登记表.md`、`05-验收标准.md` 的区别：风险登记表装的是真正的风险（PMBOK框架，可能性+影响+应对），
> 验收标准装的是"当前在做的这个阶段，做到什么程度算通过"。**本文档装的是已经讨论清楚、但明确决定不在当前阶段
> 做的产品想法/功能方向**——想法本身有价值，只是还没排进具体某个阶段。每条只记想法+为什么+前提条件，不展开设计，
> 真要排期时再单独定验收标准（照阶段六以前那些阶段的规矩）。
>
> **2026-07-19 按成本/前提条件分了三层优先级**，供下一阶段排期参考，不是绝对顺序，排期时可调整。

## 第一层：成本低，随时能排

### 划线删除功能

用户可以删除划线（不满意/不想保留的可以删掉）；问答记录不强需求要删除，暂不做对应功能。后端目前只有 `/history/{id}`（问答记录）的删除接口，**没有对应的划线删除接口**，需要新增（加一个DELETE接口+前端删除入口，技术实现本身不复杂）。

### 苏格拉底模式默认提示词/一键发送

划线后进入苏格拉底模式，输入框应该默认预填类似"讲解"这样的提示词，用户直接点发送即可，不用每次手动打字——苏格拉底模式下用户很少需要自定义追问内容，默认值能省掉大部分手动输入。

### "我的" tab 功能范围（一次讨论，不是开发工作量）

底部导航"我的"tab目前是占位页（显示"正在开发中"），长期应该放什么内容还没讨论过（账号信息？使用统计？设置项？"个人画像呈现"如果做了，可能的呈现入口之一？）。需要单独拉一次讨论明确这个入口承载什么价值。

## 第二层：有持续价值、可以现在小范围开始，但没有"做完"的终点

### 苏格拉底/讲解提示词质量优化

苏格拉底模式（尤其面对文言文/国学概念）经常直接抛出专业术语提问（比如直接问"这句话是什么意思"这类可能读者本身就不懂的概念），缺少"先确认基础理解、再由浅入深引导"的层次感，追问语言也需要更通俗。这属于持续性的提示词调优工作，不是一次性能改完的，要在后续模型/提示词迭代阶段持续打磨，排期时按"持续投入"而不是"一次性任务"来看待。

### 语音交互的手动打断按钮

点一下按钮就能打断当前AI生成/播放、立刻说新内容。这是"语音交互体验升级"（见下方第三层）里工作量最小的一个子项，可以不等完整的语音可视化界面单独先做。

## 第三层：依赖前置条件，现在不适合排

### 语音交互体验升级（可视化界面）

对话界面从"聊天气泡"升级成语音优先的可视化交互：用户说话时显示声波/水波纹动画，AI思考时有转动指示，AI回答时实时反馈；完整的自动打断（VAD语音活动检测+取消生成+重新监听）工作量较大。**前提条件**：等前端/界面设计阶段展开。

### 数字人形象

给对话界面配上具体的人物形象（比如孔子），说话时有动作/表情反馈，营造"真的在跟这个人对话"的感觉。可能需要连带更换语音模型（现在的TTS音色跟人物形象不一定匹配）。**前提条件**：依赖"语音交互体验升级"先做出来，且要等前端设计阶段才展开，是目前最长期的一条。

### 知识图谱式关联呈现

复盘页里"关联主题"板块目前是列表+文字标注的呈现方式（阶段六范围）。用户实际用起来后发现：问答数量一多、质量参差不齐时，列表形式会显得凌乱。长期设想是做成"星系/星球式"的图形结构——不同问答/划线作为节点，用连线表示语义关联。**前提条件**：阶段六的简单版关联标注先稳定跑一段、积累真实使用反馈之后再评估，不能跳过验证直接做复杂版本。

### 关联主题的 AI 生成式主题提炼

"关联主题"板块目前的命名方式是直接截取原始文字当标签。进阶版是用向量聚类+AI生成抽象主题词，产出用户自己的兴趣关键词地图。**前提条件**：同上，简单版关联标注先跑稳，不贸然在不稳的地基上加更复杂的一层。

### 划线卡片美学呈现

划线内容不只是纯列表展示，做成书签/明信片风格的可视化卡片，可收藏、可翻阅。价值主张待明确——是"做给自己看的记忆留存"还是"做给别人看的社交分享"。**前提条件**：等前端设计阶段，用户目前没有明确审美方向，计划以后借助其他模型迭代设计。

### 个人画像呈现

`highlights`/`qa_history` 积累的数据（含embedding）理论上可以反推用户的阅读兴趣/认知画像。要不要做成用户可见的界面，还是只在后台用来调整AI讲解方式，两种方向都还没定。**前提条件**：等前面几项跑起来有真实数据支撑；一旦真正启动设计，必须联动 `03-风险登记表.md` 里"用户数据使用边界"那条风险一起讨论，不能只做产品设计不管这层。

用户在另一台设备部署 [Hermes Agent](https://hermes-agent.org/)（Nous Research 开源agent运行时），设想让它和 Claude Code 共享项目文件夹、互相协商，在 Claude 卡住时充当第二大脑响应。**前提条件**：Hermes 还在部署阶段未跑通验证；两个agent同时读写同一文件夹的协作冲突机制还没设计；需要先为每个开发阶段写清楚"可能出问题的环节+应急预案+升级通知规则"。
