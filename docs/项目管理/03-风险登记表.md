# 风险登记表（Risk Register）—— 伴读讲讲手机端 v1

> 依据 INFS3703 Week 8 风险管理框架起草，按单人+AI协作项目的实际情况裁剪。

| 风险 | 类别 | 可能性 | 影响 | 应对方式 |
|---|---|---|---|---|
| WebView 渲染方案性能/桥接复杂度（AI 讲解需要跨越 EPUB 内容与 App 主体之间的桥） | 技术 | 中 | 中 | 先用简单方案（`@epubjs-react-native`）验证核心体验，真遇到卡顿再迁移到 Readium（已知的升级路径） |
| 中文公版书资源相对有限（古登堡计划中文分区数量少） | 内容 | 高 | 中 | 需要额外调研更丰富的中文公版书源 |
| 需求是伪需求，商业价值不及预期 | 业务 | 未知 | 高 | 推迟到 v1 做完后用演示视频 + 身边人反馈验证，不提前 All in |
| 不擅长自媒体/获客，不知道如何起步 | 业务 | 中 | 中 | 用"演示驱动"代替"自媒体运营"，先让产品自己说话 |
| 开发过程中在细节里打转、范围蔓延（vibe coding 老毛病） | 过程 | 中 | 中 | 靠这份 WBS 本身约束，按阶段推进不跳跃 |
| 单人开发，没有备用人力，环节卡住无人分担 | 过程/资源 | 低 | 低 | **靠日常人工监控兜底**——用户每天都会关注项目进展，问题能被及时发现，不需要现在搭建复杂的应急机制 |
| EPUB 下载接口的 token 走 URL 查询参数（`expo-file-system` 不支持自定义请求头，被迫的技术妥协）——URL 会被浏览器历史/代理/服务器日志记录，泄露面比请求头大；且当前用的是"当天有效的全接口通用 token"，命长、权大 | 安全 | 低 | 低（单用户阶段） | **2026-07-11 用户拍板：先接受现状**。单用户、无真实用户数据，泄露最坏后果是当天被人蹭接口消耗 DeepSeek 额度。**硬性触发条件：开放多用户注册之前，必须改成"短效+仅限下载"的专用 token**（类似 S3 presigned URL 的思路），不能带着这个方案上多用户 |
| `highlights`/`qa_history` 里的用户数据即便完全匿名，仍可能在用户没有合理预期的情况下被使用（如训练第三方模型、数据商业化）——项目至今没有系统讨论过这个边界 | 合规/伦理 | 未知 | 中 | **2026-07-17 用户拍板：主动延后，不是遗漏**。现在讨论意义不大，等做到相关功能设计（个人画像/数据商业化）时再深入。**硬性触发条件：设计个人画像或任何涉及用户数据对外使用/商业化的功能之前，决策层必须主动把这个话题捞回来讨论**，不能真的忘掉 |
| 苏格拉底模式缺少"这一轮知识点讲完了"的状态追踪机制，导致已经讲透的内容被重复讲解，且不会主动引导用户总结/进入下一个问题——用户亲身体验过的真实缺陷，不是假设 | 产品 | 高（已实测发生） | 中 | 便宜的短期缓解（重申系统指令+示范对话样本+生成后自我检查）可以随时做；根治需要状态追踪机制，**列为下一轮迭代候选项，不在阶段五范围内** |
