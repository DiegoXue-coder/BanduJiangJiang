# 风险登记表（Risk Register）—— 伴读讲讲手机端 v1

> 依据 INFS3703 Week 8 风险管理框架起草，按单人+AI协作项目的实际情况裁剪。

| 风险 | 类别 | 可能性 | 影响 | 应对方式 |
|---|---|---|---|---|
| WebView 渲染方案性能/桥接复杂度（AI 讲解需要跨越 EPUB 内容与 App 主体之间的桥） | 技术 | 中 | 中 | 先用简单方案（`@epubjs-react-native`）验证核心体验，真遇到卡顿再迁移到 Readium（已知的升级路径） |
| 中文公版书资源相对有限（古登堡计划中文分区数量少） | 内容 | 高 | 中 | 需要额外调研更丰富的中文公版书源 |
| 需求是伪需求，商业价值不及预期 | 业务 | 未知 | 高 | 推迟到 v1 做完后用演示视频 + 身边人反馈验证，不提前 All in |
| 不擅长自媒体/获客，不知道如何起步 | 业务 | 中 | 中 | 用"演示驱动"代替"自媒体运营"，先让产品自己说话 |
| 开发过程中在细节里打转、范围蔓延（vibe coding 老毛病） | 过程 | 中 | 中 | 靠这份 WBS 本身约束，按阶段推进不跳跃 |
| 单人开发，没有备用人力，环节卡住无人分担 | 过程/资源 | 低 | 低 | **靠日常人工监控兜底**——用户每天都会关注项目进展，问题能被及时发现，不需要现在搭建复杂的应急机制 |

## 未来可选的升级路径（不在 v1 风险应对范围内，先记一个种子）

**用 Hermes Agent 作为"第二响应层"**：用户正在另一台电脑上部署 [Hermes Agent](https://hermes-agent.org/)（Nous Research 开源 AI agent 运行时，2026 年最快增长的开源 agent 框架，原生支持 MCP 客户端 + 多智能体看板编排），设想让它和 Claude Code 共享项目文件夹、互相协商，在 Claude 卡住时充当第二大脑响应，处理不了再升级通知用户。

**前提条件（尚未满足，暂不展开设计）**：
1. Hermes 还在部署阶段，尚未实际跑通验证
2. 两个 agent 同时读写同一项目文件夹存在冲突风险，需要先解决协作机制（类似多人协作代码需要版本控制）
3. 需要为每个开发阶段写清楚"可能出问题的环节 + 应急预案 + 升级通知规则"，目前还没做

**结论**：这条先记录为未来可能的增强方向，不纳入 v1 的风险应对方案，等 Hermes 部署跑通后再单独设计。
