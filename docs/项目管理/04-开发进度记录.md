1. 开始工作前，先读这份文件的最新记录，了解上一次做到哪、留了什么问题
2. 做完一段有意义的工作（不用每个小改动都记），补一条新记录
3. **区分"必须停下等用户回复"和"可以自己判断、边记录边继续"**（2026-07-11 修订，之前的规则太保守导致项目进度依赖用户实时在场，效率上不去）：

   - **可以自己判断、不用等回复就继续做的**：技术实现细节的取舍、方案是可逆的、影响范围局限在当前这一小块功能——用自己认为最合理的方案往下做，把"做了什么判断、为什么这么判断"写清楚记在这份记录里即可，不用等用户回复才继续。用户看到记录后不同意的话，随时可以叫停或要求返工，但不需要提前等批准再动手

**关于 Railway 文件系统持久性（阶段一遗留问题）：**

**决定：现在就处理，不推迟。** 给这个服务在 Railway 后台挂一个持久卷（Volume），挂载到 `epub_storage/` 目录——这是 Railway 内置功能，不用改代码、不用换成 S3 这类对象存储。请华为那边确认一下当前 Railway 服务是否已经挂了 Volume，没有的话加上，然后验证一下重新部署后之前导入的书是否还在。

**关于 EPUB 章节切分信号（阶段二当前问题）：**

方向同意，但请先检查章节标题段落有没有 `<h1>`-`<h6>` 语义化标签或 `id` 属性——这个信号比 `margin-top` 这种 CSS 样式匹配更稳定，不会因为 Gutenberg 转换脚本换版本就失效。没有的话再退回 `margin-top` 方案。

另外：**这个切分逻辑不需要追求能应付任意未来 EPUB 的完美通用算法**——v1 书库是人工筛选的精品库，不是自动化管道，每本书上架前都会人工确认。切分完之后，请把实际渲染出来的章节结构截图/展示一下，人工确认切得对不对，而不是只看代码跑通、返回值没报错。

这次的教训（EPUB 并非都按章拆分，Gutenberg 尤其如此）已经补进 `docs/学习笔记/03-EPUB书库渲染与划线定位.md`。

## 2026-07-11（续）· 用户+这台电脑的 Claude · 章节切分方案定稿 + 一次战略考量（未采纳，记录留档）

**章节切分最终方案（级联式启发式规则，按可靠程度从高到低依次尝试，前面能确认就不用往下试）：**

1. EPUB 自带的官方目录（nav.xhtml / NCX）——最可靠，但 Gutenberg 这两本书目录本身是坏的，用不上
2. 语义化标题标签（`<h1>`-`<h6>`）或 `id` 属性——**优先尝试这个**，不会因转换脚本换版本失效
3. 命名锚点（如 `pgepubid` 这类 Gutenberg 常见的内部编号）——次选
4. CSS 样式匹配（如今天发现的 `margin-top`）——**最后的备用手段**，风险是会"悄悄切错却不报错"，不能作为唯一依据

不需要追求"能应付任意未来 EPUB 的完美算法"——v1 书库人工筛选，每本书上架前人工确认即可。以后如果书库规模扩大到人工看不过来，或者开放用户自行上传，可以参考 2026 年主流 PDF 解析工具（如 Reducto）的"两层叠加"模式：级联规则先跑一遍（快、便宜），再让 AI 检查纠正（替代人工质检），不用整个换成"AI 看图"式的重型方案——现在的问题复杂度（纯文字，无表格/多栏排版）用不上那种量级的方案。

**战略考量（讨论过，决定不采纳，留档避免以后重复讨论）：**

讨论过要不要把产品定位从"读书陪伴"扩大成"万能文档讲解"（用户上传任意 PDF/Word/图片，划选任意内容，AI 语音讲解+苏格拉底提问，跟点读笔一样）。市场潜力可能更大，且用户上传自己的文档能完全绕开 DRM 限制。**但决定暂不采纳，继续按现有"公版经典精读"方向做**，理由：

1. "万能文档讲解"赛道拥挤（Google NotebookLM 等大公司已在做），"精读经典+苏格拉底陪伴"反而是 DRM 限制意外造就的差异化窄众定位，扩得越广这个优势越稀释
2. 技术上确认了**不会造成浪费**：AI 大脑（DeepSeek 对话、语音架构、RAG、`ask_about_passage` 这类"独立能力"接口设计）完全格式无关，以后真要扩展到 PDF/图片，这层原样复用；会作废/需要重做的只是 EPUB 专属的部分——渲染方案（WebView+epub.js）、CFI 划线定位、`books`/`chapters` 这套偏"书"形状的数据库设计。这是"两条腿走路"那次决策（后端按独立能力设计，不绑定单一形态）现在开始回报的地方。

**结论：现有架构不需要为这个假设性的更大方向改动，继续推进阶段二。**

## 2026-07-13（续）· 华为服务器(Claude Code) · 真机测试补验：发现并修复了阅读器的真bug

**背景：** 阶段三验证空白（"WebView实际渲染只在web端测过、没在真机测过"）今天补上了——用户人在外面，手机没和电脑同一个WiFi，一开始想用 `expo start --tunnel`，发现 ngrok 现在强制要求注册账号才能用免费隧道（`ERR_NGROK_4018`），我不能替用户注册账号，改成让用户回家后连同一WiFi、走局域网模式，中间因为 WSL2 和 Windows 主机不是同一个网络，让用户自己在管理员权限下加了端口转发+防火墙规则（这类改系统设置的操作我不能替他做，是他自己在管理员PowerShell里跑的）。

**用户回家连上后，真机上复现了和之前web端测试一样的现象："正在下载书本"卡住不动——这次不是web平台限制了，是真的bug：**

- 查了 `@epubjs-react-native` 源码（`getSourceType.js`）才发现：这个库判断"这是不是一本epub"，靠的是**检查传入的URL字符串里有没有包含`.epub`这个子串**，不是看HTTP响应头的Content-Type。我的下载接口路径是 `/app/books/{id}/file`，压根没有`.epub`字样，导致库内部把它判定成"未知类型"、抛了个内部错误，但这个错误没有传到UI层，界面就一直卡在加载中转圈，不会提示"失败"——非常隐蔽的坑，看代码源码才找到，不是靠猜的
- 修复：接口路径改成 `/app/books/{id}/file.epub`（`api/main.py` + `mobile/lib/api.js` 同步改），本地起测试库验证过新路径能正常下载合法EPUB后才推到生产，推送时正好撞上华硕那边一条审阅提交（批准阶段三+确认token走query的取舍），已正常合并，不是冲突
- 部署验证时额外发现一个**我自己操作上的失误，记录下来避免下次重复**：本地测试鉴权token时，用命令行工具测试脚本里用了`$TOK`这类shell变量，但这类命令是通过"外层Windows Git Bash → 调用wsl.exe → 内层Ubuntu bash"两层shell转发执行的，变量插值会被外层shell提前（错误地）展开成空值，导致好几次"鉴权失败"的假象——排查了一圈以为是服务器时区不对（怀疑Railway用UTC、本地用悉尼时区，日期没对齐），后来把测试命令写成脚本文件在执行才发现是这个假警报，不是真的时区问题（服务器和本地对"今天"的判断其实是一致的）

**下一步：** 用户会在真机上继续测（长按划线、退出重进是否恢复进度、主题切换），如果一切正常，阶段三就算真正闭环了，可以开始阶段四。

## 2026-07-14（续）· 华为服务器(Claude Code) · 阶段三真机测试：又连续挖出两个真bug，现在真的能打开书了

**背景：** 上一条记录里的 `.epub` 路径修复推上去之后，用户真机重试，**还是卡在"正在下载书本"**——说明真机上还有别的问题，不是 web 端限制那种"测不到"，是真的没修完。整个排查过程记录如下，供以后遇到类似"某个RN第三方库莫名其妙卡住不动、没有任何报错"的情况参考。

**排查方法上的一个教训：** 一开始按"猜测→改代码→让用户重试"的方式来回试了好几轮都没进展（中间还包括教用户在手机上做"完全关闭App重开"这类操作，靠文字描述沟通效率很低，还生产了几次"到底有没有真的重新加载"这样的不确定性）。后来改成**直接在 `node_modules` 里的第三方库源码里插入 `console.log`**（因为 Metro 会把手机上的 `console.log` 转发回开发服务器的终端，不需要用户截图或转述），才真正定位到问题——**遇到黑盒式卡住不动又没报错的情况，插桩比反复猜测靠谱得多**。

**挖出来的两个真bug（都是 Expo SDK 54 这个版本和老一点的第三方库之间的兼容性断层，不是我们自己业务代码的问题）：**

1. **`@epubjs-react-native/expo-file-system` 这个库用的是已经被 Expo SDK 54 废弃的旧版 `expo-file-system` API**（`documentDirectory`、`writeAsStringAsync` 等）。SDK 54 把这些旧方法整体移到了 `expo-file-system/legacy` 子路径，top-level 的 `expo-file-system` 现在导入这些方法会**直接抛错**（内容就是那句反复出现的"deprecated, migrate to..."提示，这也是为什么日志里一直看到这条"警告"其实是被吞掉的报错）。库本身没针对SDK 54更新。修复：把这个库内部的 `require("expo-file-system")` 改成 `require("expo-file-system/legacy")`，恢复成能正常工作的旧API。
2. **修完第1个坑后，书本内容开始真的往前推进，但又冒出一个新的、真机上能看见的红屏报错**：`RNGestureHandlerModule...configureRelation is not a function`。查出来是 `react-native-gesture-handler` 被 npm 自动装成了最新的 3.0.2（因为 `@epubjs-react-native/core` 只声明了 `"*"` 这种任意版本的依赖关系，没锁定），但 Expo Go 在 SDK 54 下内置的原生模块只兼容 2.x 系列，版本对不上导致某个原生方法在JS侧调不到。修复：用 `npx expo install react-native-gesture-handler` 装成SDK匹配的 `~2.28.0`；同时按 `react-native-gesture-handler` 官方要求，在 `App.js` 根节点包了一层 `<GestureHandlerRootView>`（这个库用的是新版 Gesture 组合式API，理论上必须包这一层手势才能正常注册，之前没包是因为项目原来没有真正用到手势交互）。

**持久化处理：** 因为 `node_modules` 不进 git，第1个坑的修复用 `patch-package` 生成了补丁文件（`mobile/patches/@epubjs-react-native+expo-file-system+1.1.4.patch`），并在 `package.json` 加了 `"postinstall": "patch-package"`，验证过重新 `npm install` 后补丁会自动重新应用——不然华硕那边或者任何人重新装一遍依赖，这个坑会原样复现。

**验证：** 用户真机（iPhone，Expo Go）上实测确认这本书的内容现在能正常加载出来了。至此阶段三"WebView实际渲染在真机上没验证过"这个遗留空白正式补上，且过程中修复的两个bug都是会影响所有用户、所有书本的通用性问题（不是某本书特有），修复具有普遍性。

**下一步：** 阶段四——AI 对话 + 语音（复用已验证过的 `ReaderChatScreen.js` 管线，改造成读取自己书库的书本上下文）。
