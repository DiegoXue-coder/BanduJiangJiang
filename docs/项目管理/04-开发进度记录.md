## 手机真机连接怎么起（每次新会话/换网络都要走一遍）

> 新对话/新设备打开这个项目、需要真机测试时，直接照这个流程操作，不用重新摸索。
> 现在有两条路，看场景选：**开发调试**（要改代码、要热更新、要看 console 日志）走 A；
> **纯演示/随时打开看效果**（不用同WiFi、不需要我这边开着机器）走 B。

### A. 本地开发服务器（调试用，依赖当前会话/同WiFi）

1. **启动开发服务器**：项目根目录跑 `.claude/launch.json` 里的 `mobile-lan` 配置（本质是 `npm --prefix mobile run lan`，等价于 `mobile/` 目录下 `npm run lan` = `expo start --lan`）。**不要用 `npx expo start`**——版本跟项目里锁定的不一致，会导致 Expo Go 报"无法连接"。
2. **必须用局域网模式，不用隧道模式**：`npm run tunnel` 在国内网络下经常连不上 `exp.direct` 这个域名（哪怕电脑本身有 VPN/Tailscale 能正常上网，也不代表隧道服务能连通），只在局域网模式完全不可用时才退回去试。
3. **前提**：手机和电脑必须连同一个 WiFi（不能用手机流量）。换了网络（换 WiFi、回家/换地方）之后，电脑的局域网 IP 会变，要重新走第4步查一遍。
4. **查电脑当前局域网 IP**：`ipconfig`，找"无线局域网适配器 WLAN"下面的 IPv4 地址（不是以太网那个）。
5. **生成/更新连接二维码**：把 IP 拼成 `exp://<IP>:8081`，做成二维码图片（发 Artifact 或图片都行）给用户用手机相机扫（不用先打开 Expo Go，扫完系统会提示用 Expo Go 打开）。**只有 IP 真的变了才需要重新生成**——服务器重启但 IP 没变的话，旧二维码/旧链接直接复用。
6. **验证服务器活着**：`curl http://<IP>:8081/status`，200 就说明没问题。
7. 这个开发服务器是当前会话/沙箱进程的产物，**换一个新对话或沙箱重启后进程会没掉**，需要重新走第1步启动，其余步骤视 IP 是否变化决定要不要重做。

### B. EAS Update 云端发布（演示用，不依赖同WiFi、不依赖我这边的会话）

**2026-07-21 新增**（起因：用户嫌每次对进度都要让我重启本地服务器太麻烦，还想能在外面拿给朋友演示）。项目已经跑过 `eas init` + `eas update:configure`，链接到了 `@diegoxue/bandujiangjiang` 这个 EAS 项目（`mobile/app.json` 里的 `extra.eas.projectId`，已提交）。

1. **发布一版**（每次代码有更新、想让手机上看到最新效果时执行）：
   ```
   cd mobile && npx eas-cli update --branch preview --message "简短说明这次改了什么" --non-interactive
   ```
   输出里会有一个 `Update group ID`，记下来。
2. **生成扫码链接**（不需要登录，公开可访问）：
   ```
   curl "https://qr.expo.dev/eas-update?projectId=7284c7eb-916b-4d0f-9392-52ef9a683c48&groupId=<上一步的Update group ID>" -o qr.svg
   ```
   纯文本链接（不生成图片、只要文字发给用户自己转二维码时用）加 `&format=url` 参数。
3. **原理**：这个链接指向 Expo 的云端（`u.expo.dev`），用户拿 Expo Go 扫码时是直接从 Expo 服务器拉取代码运行，全程不经过我这台机器，因此不需要同一个WiFi、也不需要我这边开着某个会话/进程。
4. **首次踩坑记录**：Expo 官网 `expo.dev/preview/update` 这个网页版预览页面，选择"Expo Go"那个子页面本身需要登录态才能看，浏览器沙盒里没有用户的登录session，走不通；改成上面这个 `qr.expo.dev/eas-update` 公开端点直接拿链接，不用登录，親測有效。
5. **局限**：这条路只能更新 JS/资源这一层（"OTA更新"），如果以后加了 Expo Go 本身不自带的原生模块，这条路就不够用了，需要换成 `eas build` 出一个独立 App。

## 2026-07-11（续）· 用户+这台电脑的 Claude · 章节切分方案定稿 + 一次战略考量（未采纳，记录留档）

**章节切分最终方案（级联式启发式规则，按可靠程度从高到低依次尝试，前面能确认就不用往下试）：**

1. EPUB 自带的官方目录（nav.xhtml / NCX）——最可靠，但 Gutenberg 这两本书目录本身是坏的，用不上
2. 语义化标题标签（`<h1>`-`<h6>`）或 `id` 属性——**优先尝试这个**，不会因转换脚本换版本失效
3. 命名锚点（如 `pgepubid` 这类 Gutenberg 常见的内部编号）——次选
4. CSS 样式匹配（如今天发现的 `margin-top`）——**最后的备用手段**，风险是会"悄悄切错却不报错"，不能作为唯一依据

不需要追求"能应付任意未来 EPUB 的完美算法"——v1 书库人工筛选，每本书上架前人工确认即可。以后如果书库规模扩大到人工看不过来，或者开放用户自行上传，可以参考 2026 年主流 PDF 解析工具（如 Reducto）的"两层叠加"模式：级联规则先跑一遍（快、便宜），再让 AI 检查纠正（替代人工质检），不用整个换成"AI 看图"式的重型方案——现在的问题复杂度（纯文字，无表格/多栏排版）用不上那种量级的方案。

**战略考量（讨论过，决定不采纳，留档避免以后重复讨论）：**

讨论过要不要把产品定位从"读书陪伴"扩大成"万能文档讲解"（用户上传任意 PDF/Word/图片，划选任意内容，AI 语音讲解+苏格拉底提问，跟点读笔一样）。市场潜力可能更大，且用户上传自己的文档能完全绕开 DRM 限制。**但决定暂不采纳，继续按现有"公版经典精读"方向做**，理由：

1. "万能文档讲解"赛道拥挤（Google NotebookLM 等大公司已在做），"精读经典+苏格拉底陪伴"反而是 DRM 限制意外造就的差异化窄众定位，扩得越广这个优势越稀释
2. 技术上确认了**不会造成浪费**：AI 大脑（DeepSeek 对话、语音架构、RAG、`ask_about_passage` 这类"独立能力"接口设计）完全格式无关，以后真要扩展到 PDF/图片，这层原样复用；会作废/需要重做的只是 EPUB 专属的部分——渲染方案（WebView+epub.js）、CFI 划线定位、`books`/`chapters` 这套偏"书"形状的数据库设计。这是"两条腿走路"那次决策（后端按独立能力设计，不绑定单一形态）现在开始回报的地方。

**结论：现有架构不需要为这个假设性的更大方向改动，继续推进阶段二。**

## 2026-07-13（续）· 华为服务器(Claude Code) · 真机测试补验：发现并修复了阅读器的真bug

**背景：** 阶段三验证空白（"WebView实际渲染只在web端测过、没在真机测过"）今天补上了——用户人在外面，手机没和电脑同一个WiFi，一开始想用 `expo start --tunnel`，发现 ngrok 现在强制要求注册账号才能用免费隧道（`ERR_NGROK_4018`），我不能替用户注册账号，改成让用户回家后连同一WiFi、走局域网模式，中间因为 WSL2 和 Windows 主机不是同一个网络，让用户自己在管理员权限下加了端口转发+防火墙规则（这类改系统设置的操作我不能替他做，是他自己在管理员PowerShell里跑的）。

**用户回家连上后，真机上复现了和之前web端测试一样的现象："正在下载书本"卡住不动——这次不是web平台限制了，是真的bug：**

- 查了 `@epubjs-react-native` 源码（`getSourceType.js`）才发现：这个库判断"这是不是一本epub"，靠的是**检查传入的URL字符串里有没有包含`.epub`这个子串**，不是看HTTP响应头的Content-Type。我的下载接口路径是 `/app/books/{id}/file`，压根没有`.epub`字样，导致库内部把它判定成"未知类型"、抛了个内部错误，但这个错误没有传到UI层，界面就一直卡在加载中转圈，不会提示"失败"——非常隐蔽的坑，看代码源码才找到，不是靠猜的
- 修复：接口路径改成 `/app/books/{id}/file.epub`（`api/main.py` + `mobile/lib/api.js` 同步改），本地起测试库验证过新路径能正常下载合法EPUB后才推到生产，推送时正好撞上华硕那边一条审阅提交（批准阶段三+确认token走query的取舍），已正常合并，不是冲突
- 部署验证时额外发现一个**我自己操作上的失误，记录下来避免下次重复**：本地测试鉴权token时，用命令行工具测试脚本里用了`$TOK`这类shell变量，但这类命令是通过"外层Windows Git Bash → 调用wsl.exe → 内层Ubuntu bash"两层shell转发执行的，变量插值会被外层shell提前（错误地）展开成空值，导致好几次"鉴权失败"的假象——排查了一圈以为是服务器时区不对（怀疑Railway用UTC、本地用悉尼时区，日期没对齐），后来把测试命令写成脚本文件在执行才发现是这个假警报，不是真的时区问题（服务器和本地对"今天"的判断其实是一致的）

**排查方法上的一个教训：** 一开始按"猜测→改代码→让用户重试"的方式来回试了好几轮都没进展（中间还包括教用户在手机上做"完全关闭App重开"这类操作，靠文字描述沟通效率很低，还生产了几次"到底有没有真的重新加载"这样的不确定性）。后来改成**直接在 `node_modules` 里的第三方库源码里插入 `console.log`**（因为 Metro 会把手机上的 `console.log` 转发回开发服务器的终端，不需要用户截图或转述），才真正定位到问题——**遇到黑盒式卡住不动又没报错的情况，插桩比反复猜测靠谱得多**。

**挖出来的两个真bug（都是 Expo SDK 54 这个版本和老一点的第三方库之间的兼容性断层，不是我们自己业务代码的问题）：**

1. **`@epubjs-react-native/expo-file-system` 这个库用的是已经被 Expo SDK 54 废弃的旧版 `expo-file-system` API**（`documentDirectory`、`writeAsStringAsync` 等）。SDK 54 把这些旧方法整体移到了 `expo-file-system/legacy` 子路径，top-level 的 `expo-file-system` 现在导入这些方法会**直接抛错**（内容就是那句反复出现的"deprecated, migrate to..."提示，这也是为什么日志里一直看到这条"警告"其实是被吞掉的报错）。库本身没针对SDK 54更新。修复：把这个库内部的 `require("expo-file-system")` 改成 `require("expo-file-system/legacy")`，恢复成能正常工作的旧API。
2. **修完第1个坑后，书本内容开始真的往前推进，但又冒出一个新的、真机上能看见的红屏报错**：`RNGestureHandlerModule...configureRelation is not a function`。查出来是 `react-native-gesture-handler` 被 npm 自动装成了最新的 3.0.2（因为 `@epubjs-react-native/core` 只声明了 `"*"` 这种任意版本的依赖关系，没锁定），但 Expo Go 在 SDK 54 下内置的原生模块只兼容 2.x 系列，版本对不上导致某个原生方法在JS侧调不到。修复：用 `npx expo install react-native-gesture-handler` 装成SDK匹配的 `~2.28.0`；同时按 `react-native-gesture-handler` 官方要求，在 `App.js` 根节点包了一层 `<GestureHandlerRootView>`（这个库用的是新版 Gesture 组合式API，理论上必须包这一层手势才能正常注册，之前没包是因为项目原来没有真正用到手势交互）。

**持久化处理：** 因为 `node_modules` 不进 git，第1个坑的修复用 `patch-package` 生成了补丁文件（`mobile/patches/@epubjs-react-native+expo-file-system+1.1.4.patch`），并在 `package.json` 加了 `"postinstall": "patch-package"`，验证过重新 `npm install` 后补丁会自动重新应用——不然华硕那边或者任何人重新装一遍依赖，这个坑会原样复现。

**验证：** 用户真机（iPhone，Expo Go）上实测确认这本书的内容现在能正常加载出来了。至此阶段三"WebView实际渲染在真机上没验证过"这个遗留空白正式补上，且过程中修复的两个bug都是会影响所有用户、所有书本的通用性问题（不是某本书特有），修复具有普遍性。

## 2026-07-18（续）· 用户+华硕电脑的 Claude · 关联主题设计改进（3轮迭代）

用户看了上面"4条关联"的实际内容，提出设计层面的问题：应该按划线内容（selection）关联更合理，现在还是有问题。三轮修复：

**第一轮——embedding 公式漏了 selection：** 之前算向量只用 `question+answer`，没用 `selection`（划线原文）。问题信息量低（"这个是什么意思"）、苏格拉底模式回答又短的情况下，向量基本是在比较"措辞像不像"，跟真实内容无关，这是关联检测把不相关记录标"相关"的根本原因。**改动**：公式改成 `selection+question+answer`（原文压过空洞问题的干扰，同时不丢失发现跨书概念联系的能力——纯比 selection 会漏掉"文字不同但主题相通"的情况，比如道德经"不争"和论语"君子无所争"）。新增一次性维护接口 `POST /app/backfill-embeddings`，把现有29条记录全部用新公式重新算了一遍向量（**这一步之前用户已经提醒过优先级，但上一轮我漏做了，这次一起补上**）。生产验证：关联数从19条降到12条。

**第二轮——排除自我关联的过滤器不够严格：** 12条里发现好几条其实还是同一段原文的自我关联，只是没被"排除 selection 相同"那条挡住——同一段论语原文，有的记录划选时带了"15. "这种章节编号前缀、有的没带，字符串不完全相等。**改动**：过滤条件加 `regexp_replace` 去掉开头的编号前缀再比较。生产验证：12条降到2条。

**第三轮——标注文字本身还是没有分析价值（用户截图指出）：** 剩下的2条匹配本身是对的（内容确实沾边），但标注显示的是对方那条记录的**提问内容**（"你好"、"这个什么意思"），提问本身信息量低，就算匹配对了，用户也看不出"关联"具体指什么。**改动**：`related_question`（对方的提问）改成 `related_text`（对方的划线原文，超过30字截断）。效果："与《道德经》里的'你好'相关" → "与《道德经》里的'天下'相关"，现在用户能一眼看出两条记录是因为都涉及"天下"这段内容才关联上的。

## 2026-07-19 · 用户+华硕电脑的 Claude · 修复"跳转到原文位置"在书已打开时不生效

用户对照阶段六功能清单验收，发现"点击复盘卡片详情跳回原文具体位置"这项没做到。排查：

**根因：** `initialLocation` 这个属性（多数 epub 阅读器组件的通用做法）只在组件**第一次挂载**时生效。用户如果已经打开过某本书（Reader 页面还挂在书架堆栈里没被销毁），从划线复盘详情页点"跳转到原文位置"时，`navigation.navigate` 只是更新了已挂载页面的 route.params，不会触发重新挂载——`initialLocation` 的新值传进去了，但底层阅读器根本不会重新读这个属性，跳转没有任何效果。这是最常见的实际使用场景（先读书划线，之后去复盘页回看），之前只测了"书还没打开过"这一种情况，没测到这个。

## 2026-07-21 · 用户+华硕电脑的 Claude · 苏格拉底模式提示词补两个缺口

2. 第二轮和第三轮及以后：都新增一条判断分支——用户回答明确是"不懂/看不懂/什么意思/不知道"这类直接求助信号时，必须以固定前缀`"先说清楚——"`开头给出2-3句有信息量的真实解释，禁止空话搪塞；不是这个信号时维持原有的点评+追问（或"你已经推导出来了"揭示洞见）逻辑。

**改动（`mobile/screens/BookChatScreen.js`）：** 新增 `preparedRef` 保存"提前加载好、还没播放"的下一句 Sound 对象；`playNextInQueue` 里一开始播放当前句，不等它播完就立刻调用新增的 `prefetchNext()` 后台加载队列里的下一句；下一句真正要播放时优先用 `preparedRef` 里已经准备好的，没有的话（比如第一句，或者上一句放得比预取还快）才现场加载，退化成原来的行为不会卡死。`stopAudio()` 顺手把预取了但没播放的音频也一并释放，不留资源泄漏。

## 2026-07-22（续）· 用户+华硕电脑的 Claude · TTS预取修复上线即出新bug：好几句声音同时播——竞态条件排查+修复

用户真机测完上面那版预取修复，反馈"好多声音重叠在一起"——这是我这次改动自己引入的回归，不是操作问题。

**根因：** `playNextInQueue` 里"标记正在播放"（`ttsPlayingRef.current = true`）这一步，被我错误地挪到了 `await Audio.Sound.createAsync(...)` 之后才执行。而 `flushSentences` 处理流式回答时，一次可能连续攒出好几句完整的话，会在同一个事件循环 tick 里连续同步调用好几次 `enqueueTts`。等待音频加载的这段 await 期间，`ttsPlayingRef.current` 还是 false，后面几次 `enqueueTts` 调用就会误判"现在没在播"，各自又触发一次 `playNextInQueue`——最终好几句音频各自调用了 `playAsync()`，同时播了出来。原来（预取修复之前）的旧版本这一步锁是在 shift 完之后立刻同步设置、await 之前，没有这个窗口期，这次改动把这层保护弄丢了。

**修复：** 把"占用要播的这一句 + 上锁"这两步挪回全部同步完成（不能插入任何 await），跟原来旧版本的写法一个道理。同时发现 `prefetchNext()` 也有一个同构的竞态（虽然后果不是重叠播放，是静默丢句子——两次并发预取会互相覆盖 `preparedRef`），新增 `preparingRef` 这个锁一并堵上。

**教训：** 引入"提前加载、延后处理"这类异步优化时，原来那个"锁必须在第一个 await 之前同步上"的约束容易在重构时被无意间破坏——这次这个坑，是把原本紧挨着写的"shift + 上锁"两行拆开、中间插进了一段 `await` 造成的。以后碰到类似的队列/锁改造，改完要专门检查一遍"锁上锁的时机相对 await 有没有移动过"。

**2. 跨书关联检测自动化校验工具——工具写完了，但卡在权限，没能实际跑。**

新增 `api/eval/relation_validation.py`：用 DeepSeek 生成候选跨书文字对（一半"明确应该关联"、一半"明确不应该关联"，锚定在书库里真实存在的书），通过真实生产接口（`POST /app/books/{id}/highlights`、`POST /history`）存成真实的划线+问答记录——跑完直接打开App复盘页就能亲眼验证当前阈值准不准，不是看一份后台报告，符合验收标准里"必须能在App里被用户亲眼验证"这条要求。

**卡住的地方：** 这个工具要往生产数据库真实写数据，本地这台机器两条路都走不通——① 本地网络连不上生产 Postgres（今天早些时候起本地 uvicorn 服务器就撞过这个问题，环境限制，不是这次新发现的）；② 调用生产接口需要 `EXTENSION_SECRET`（跟手机 App 用的是同一套 HMAC 日签名鉴权），这个值只配置在 Railway 生产环境，本地 `api/.env` 里没有。

已经验证过工具本身逻辑没问题：`--dry-run` 模式（只需要本地已有的 `DEEPSEEK_API_KEY`，不碰生产库）跑通了，生成的候选对质量总体不错（4组里3组关联点判断准确，1组略牵强但不算错）。

**下一步：** 需要用户决定给不给本地补上 `EXTENSION_SECRET`（这样以后类似的真实数据校验工具都能在这边直接跑），还是这次就用户自己在能连上生产环境的地方跑一次这个脚本。
